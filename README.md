## TLDR（一句话摘要）

一款**完全被动记录**、**隐私全本地**、**一键分享美观路线图**的旅游打卡 PWA。用户只需打开 App，系统自动追踪 GPS 轨迹、识别停留打卡点、绘制实时路径。用户**唯一主动操作**是在分享截图时为自动识别出的打卡点写备注。技术栈：Leaflet + OpenStreetMap + Chart.js + Dexie.js + Open-Meteo + Workbox，零后端依赖。

## 第一章：项目概述与设计原则

### 1.1 一句话愿景

一款完全被动记录、旅行结束后一键分享的旅游打卡路线 App。

### 1.2 三个核心设计原则

| 原则 | 含义 | 设计影响 |
|------|------|----------|
| **被动无感** | 用户不需要进行任何操作，App 在后台自动持续追踪位置 | 无开始/停止按钮，无弹窗或气泡提示，权限首次静默请求 |
| **隐私优先** | 所有用户数据存储在本地设备，不上传任何服务器 | 纯前端架构，IndexedDB 本地存储，不引入第三方 SDK（除地图瓦片和天气 API 外） |
| **旅游打卡** | 核心场景是记录旅行路线和打卡点，一键分享美观的路线图 | 自动识别停留点作为打卡点，9:16 竖版分享卡片，POI 语义化显示 |

### 1.3 技术栈总览

| 模块 | 选型 | 理由 |
|------|------|------|
| 地图引擎 | Leaflet.js 1.9+ | 免费无 API Key，轻量（~40KB），iPhone 性能良好 |
| 地图瓦片 | OpenStreetMap 标准瓦片 + 卫星瓦片图层切换 | 全球覆盖，零费用。卫星图作为国内精度补充 |
| 图表库 | Chart.js 4.x | Canvas 渲染，iPhone 性能好，支持折线图/柱状图/环形图/饼图 |
| 天气数据 | Open-Meteo API（免费，无需 Key） | 全球覆盖，支持温度/体感/AQI/UV/风速/日出日落 |
| 本地存储 | Dexie.js（IndexedDB 封装） | ~10KB，Promise API，支持复杂查询和索引 |
| PWA | Workbox（Service Worker 缓存策略） | 管理离线缓存和资源策略 |
| 截图 | html2canvas（仅用于分享卡片） | Canvas 截取地图视口，合成分享卡片 |
| 逆向地理编码 | Nominatim API（免费，无需 Key） | 坐标 → 街道/区/城市/地标 语义化，结果缓存本地 |

### 3.1 架构总览

```
┌─────────────────────────────────┐
│         用户界面 (HTML/CSS)       │
├─────────────────────────────────┤

│  · 定位管理器 (Geolocation API)  │
│  · 轨迹引擎 (点采集+降采样)      │
│  · 停留检测 (打卡点自动识别)      │
│  · POI 反向解析 (Nominatim)      │
│  · 天气查询 (Open-Meteo)         │
│  · 图表生成 (Chart.js)           │
│  · 卡片生成 (html2canvas)        │
├─────────────────────────────────┤
│         数据层 (Dexie.js)        │
│  tracks  │ trackPoints          │
│  checkinPoints │ goals           │
│  nominatimCache                  │
├─────────────────────────────────┤
│    Service Worker (PWA 离线)     │
│    Workbox 缓存策略 + 后台保活    │
└─────────────────────────────────┘
```

### 3.2 核心数据流

```
系统启动
  │
  ├─→ 静默请求定位权限（首次）
  │
  ├─→ 获取当前位置 → 地图居中
  │
  ├─→ watchPosition 持续监听（精度优先, distanceFilter=2m）
  │     │
  │     ├─→ 精度 > 100m → 丢弃该点
  │     ├─→ 距离上次点 < 3m → 跳过（去重）
  │     └─→ 合格点 → 写入 trackPoints + 更新 polyline
  │
  ├─→ 每 60 秒执行一次停留检测
  │     │
  │     ├─→ 速度 < 1km/h 且 停留 ≥ 8min → 标记为打卡点
  │     │     └─→ 异步 Nominatim 解析 → 缓存结果
  │     └─→ 否则 → 无操作
  │
  ├─→ 每 5 分钟（或新城市时）获取天气
  │     └─→ Open-Meteo → 更新首页方框
  │
  └─→ 轨迹分段判断（每 detectStop 检查）
        └─→ GPS 中断 ≥ 30min → 自动结束当前轨迹
              └─→ 写入 tracks 表 + 计算统计摘要
```

### 3.4 后台策略

**核心问题：** iOS 对 Web App（PWA）后台定位支持极弱，`watchPosition` 在切后台后约 30 秒被系统冻结。

**分级策略：**

| 级别 | 方案 | 适用场景 | 限制 |
|------|------|----------|------|
| Level 1 | 前台 `watchPosition` + `distanceFilter=2m` | App 在前台使用时 | 切后台后 30s 失效 |
| Level 2 | PWA 无声音频保活（`AudioContext` 播放静音轨道） | App 添加到主屏幕后 | 维持约 3-5min 后台活跃 |
| Level 3 | 冷启动轨迹虚线补全 | 重新打开 App 时 | 对中断段用灰色虚线表示推测路径 |

**冷启动恢复逻辑：**
1. 新打开的 App 检测上次定位时间
2. 若距离上次定位 > 30min，认为"一次旅行结束"
3. 不补画实线轨迹，只显示灰色虚线连接中断点
4. 虚线两端标注时间戳，用户点击虚线可看到"推测路径"提示文本

## 第四章：数据库 Schema

### 4.1 tracks（轨迹表）

```typescript
interface Track {
  id: number;               // 自增主键
  date: string;             // YYYY-MM-DD，非唯一索引
  startTime: number;        // 开始时间戳
  endTime: number;          // 结束时间戳
  totalDistance: number;    // 总距离（公里）
  totalDuration: number;    // 总时长（秒）
  pointCount: number;       // 路径点数量
  cityDetected: string;     // 自动识别的城市名
  region: string;           // 区/县名
  routeType: 'walk' | 'hike' | 'mixed' | 'unknown'; // 路线类型
  isManual: boolean;       // false（被动自动追踪，保留字段）
  createdAt: number;        // 创建时间戳
}
```

**索引：** `date`, `startTime`, `endTime`

### 4.2 trackPoints（路径点表）

```typescript
interface TrackPoint {
  id: number;               // 自增主键
  trackId: number;          // 关联 tracks.id，非唯一索引
  lat: number;              // 纬度
  lng: number;              // 经度
  timestamp: number;        // 采样时间戳
  accuracy: number;         // 定位精度（米）
}
```

**索引：** `trackId`, `timestamp`

**采样策略：** 写入时按原始频率写入。查询时按时间窗口做降采样展示。

### 4.3 checkinPoints（打卡点表）

```typescript
interface CheckinPoint {
  id: number;               // 自增主键
  trackId: number;          // 关联 tracks.id，非唯一索引
  pointId: number;          // 关联 trackPoints.id（最近的路径点）
  lat: number;              // 纬度
  lng: number;              // 经度
  name: string;             // 自动识别的地点名称（如"朝阳公园"）
  address: string;          // 自动识别的地址
  city: string;             // 城市名
  category: 'attraction' | 'restaurant' | 'hotel' | 'transport' | 'shop' | 'other'; // 自动分类
  note: string;             // 用户备注（仅在分享截图时填写）
  arrivedAt: number;        // 到达时间戳
  duration: number;         // 停留时长（秒）
  createdAt: number;        // 创建时间戳
}
```

**索引：** `trackId`, `category`

**自动检测逻辑：** 路径点移动速度 < 1km/h 且 持续 ≥ 8 分钟 → 标记为候选打卡点 → Nominatim 解析名称 → 写入 checkinPoints。

### 4.4 goals（目标表）

```typescript
interface Goal {
  id: number;               // 固定为 1（单条记录）
  type: 'daily_distance' | 'weekly_active_days';  // 目标类型
  target: number;           // 目标值（daily: 公里, weekly: 天数）
  createdAt: number;
  updatedAt: number;
}
```

**默认值：** `daily_distance` / 3km 或 `weekly_active_days` / 5天

### 4.5 nominatimCache（反向地理编码缓存表）

```typescript
interface NominatimCache {
  id: number;
  latKey: string;           // 经度范围 key（latMin_latMax_lonMin_lonMax）
  data: object;             // 缓存的地址解析结果
  expiresAt: number;        // 过期时间（7 天）
  createdAt: number;
}
```

**缓存策略：** 同一矩形范围（约 1km²）的坐标不重复请求。请求间隔 ≥ 1 秒（遵守 Nominatim usage policy）。缓存 TTL = 7 天。

## 第五章：功能规格 — 首页 Tab

### 5.1 首页布局

```
┌─────────────────────┐  ← env(safe-area-inset-top)
```

### 5.2 地图

- **默认视口：** 首次加载以用户当前位置为中心，缩放级别 15
- **自适应缩放：** 如果今日有轨迹 → `fitBounds` 使轨迹完整显示在屏幕内（padding 50px）
- **轨迹绘制：** 渐变线样式（起点淡入 opacity:0.4 → 终点 opacity:0.9），线宽 3-4px，颜色 #FF8C42
- **起点标记：** 空心圆点，白色边框 #FF8C42 填充
- **终点标记：** 实心圆点，白色边框 #FF8C42 填充
- **打卡点标记：** 气泡针样式——细线 + 圆形气泡 + 序号数字，颜色按分类
- **地图交互：** 支持缩放（12-18级）、拖动、双击放大
- **图层切换：** 右上角图层按钮，切换标准 OSM / 卫星图

### 5.3 地图跟随

- 用户拖动地图后 → 跟随模式关闭（右上角显示"跟随"按钮）
- 点击"跟随"按钮 → 地图回到当前位置并继续跟随
- 用户没有操作时 → 始终跟随当前位置

### 5.4 信息方框

| 框 | 内容 |
|----|------|
| 主框（宽 2 倍） | 当前城市/区域名 + 天气图标 + 温度（当前/最高/最低） |
| 辅框 1 | 今日行走距离（km）+ 今日探索打卡点数量 |
| 辅框 2 | 当前位置街道/景点名 + 备注 |

- 天气数据来源：Open-Meteo API（需定位成功后再请求，时序依赖）
- 定位失败时主框显示"位置未授权"，不显示天气
- 所有方框使用纯白卡片 + 轻阴影（`background: #FFFFFF; box-shadow: 0 2px 16px rgba(0,0,0,0.06)`）

- **点击日期的交互：** 先显示"预览卡片"（日期+星期+总距离+经-过的区域数+打卡点数），再点击预览卡片上的"查看轨迹"按钮展开全屏地图

### 6.3 当日轨迹查看页

- 全屏地图，展示该日所有轨迹段
- 轨迹线：梯度颜色（按时间从浅到深）
- 打卡点：气泡针 + 序号，点击显示轻量卡片（名称、到达时间、备注）

- 右下角缩放控件

### 6.4 方框内容

- 当日总行走距离（km）
- 活跃时段（例如"09:30-11:45, 14:00-16:20"）
- 打卡点数量
- 经过的区域数（如"3 个景区、2 条街道"）

**下半部分（环境上下文）：**
- 当日天气概况（最高/最低温、天气状况）
- 日出/日落时间
- 当日平均气温

## 第九章：自动打卡点识别

### 9.1 停留检测算法

这是一个纯客户端的自动算法，用户无感知运行。

**触发条件：** 每隔 60 秒执行一次

**检测逻辑：**

```
1. 取最近一段连续路径点（从开始到现在）
2. 计算每个点的瞬时速度 = 距离(prev, curr) / 时间差(prev, curr)
3. 标记"低速段"：速度 < 1km/h 的连续点序列
4. 计算低速段时长 = 序列最末时间 - 序列起始时间
5. 若低速段 ≥ 8 分钟（480 秒）→ 标记为候选打卡点
6. 以低速段中位数坐标作为打卡点坐标
7. 调用 Nominatim 反向地理编码 → 获取地点名称
8. 根据时段和位置类型自动分类（attraction / restaurant / ...）
9. 写入 checkinPoints 表
```

**防重复：** 同一坐标 500m 范围内 24 小时内不重复标记。

### 9.2 用户对打卡点的编辑权

打卡点虽由算法自动生成，但用户在分享预览界面拥有最终控制权：
- 点击打卡点气泡上的 ✕ 按钮 → 从当前分享卡片中删除该打卡点
- 拖拽调整打卡点显示顺序
- 为打卡点添加文字备注

**重要原则：** 分享预览中的删除操作不影响 IndexedDB 中存储的原始轨迹数据，仅影响本次生成的分享卡片。如需永久删除某条轨迹的打卡点，用户需删除整条轨迹。

### 9.3 地点分类逻辑

| 分类 | 判断依据 |
|------|----------|
| attraction | 地址包含"公园""博物馆""景区""古迹""广场"等 |
| restaurant | 地址包含"餐厅""馆""店""咖啡""小吃"等，或时段为 11:30-13:00 / 17:30-19:30 |
| hotel | 地址包含"酒店""客栈""民宿""青旅"等，且停留 > 2 小时 |
| transport | 地址包含"站""机场""出口""停车场"等 |
| shop | 地址包含"商场""超市""商店"等 |
| other | 无法自动分类的兜底 |

### 9.3 轨迹分段逻辑（旅行起止判断）

```
连续移动 → GPS 中断 ≥ 30 分钟 → 触发分段
每次触发分段的动作：
1. 结束当前轨迹：写入 tracks 表（计算距离/时长/城市等摘要）
2. 开始新轨迹：新建 trackPoints 集合，新轨迹 ID
3. 分段后恢复记录的轨迹用灰色虚线连接中断部分
```

单日可以有多个轨迹（只要中断达成 30 分钟标准）。

## 第十一章：边界场景与错误状态

### 11.1 错误状态矩阵

| 场景 | 表现 | 行为 |
|------|------|------|
| GPS 信号弱 | accuracy > 100m | 丢弃低精度点；顶栏显示淡黄条"定位信号弱"；记录频率降至 15s/次 |
| 无网络（离线） | 天气 API 请求失败 | 天气信息显示"离线模式"，灰色占位符；地图瓦片用缓存；Nominatim 跳过 |
| 存储空间满 | IndexedDB 写入失败 | 触发自动压缩：>30天前的轨迹降采样至 1/5 密度；提示"数据较多，建议导出后清理" |
| 写入失败 | Dexie 抛异常 | 温和提示"存储异常，数据可能不完整"；引导用户导出后清除 |
| 无效坐标 | 坐标为 0,0 或 null | 直接丢弃，不写入存储，不记录到数组 |
| 定位拒绝 | 用户拒绝定位权限 | 首页地图显示当前位置不可用；天气显示"位置未授权"；不记录轨迹；不重复弹权限窗 |
| 定位中途恢复 | GPS 从弱到强 | 自动恢复记录频率，路径点继续追加；中断段用灰色虚线 |
| 数据导入冲突 | 导入的轨迹日期已存在 | 根据日期去重，不覆盖已有数据 |

### 11.2 存储满自动压缩

**触发条件：** IndexedDB 可用空间 < 10MB 或写入失败

**压缩逻辑：**
1. 查询 30 天前的轨迹
2. 对每条轨迹的 trackPoints 进行降采样（每 5 个点保留 1 个）
3. 更新 track 的 pointCount
4. 发出通知："已自动优化存储空间"
5. 如果压缩后仍不足 → 引导用户导出后清除

### 11.3 空状态设计

| 页面 | 空状态显示 |
|------|------------|
| 首页（无轨迹） | 地图显示当前位置，方框正常，不显示轨迹线 |
| 记录 Tab（无数据） | 日历无标记，"尚无足迹记录"提示 |
| 分析 Tab（数据<3天） | "收集更多足迹，分析更有趣" |
| 分析 Tab（当月无数据） | "本月还没有记录" |
| APP Tab（无数据） | 正常显示，可导出空数据文件 |

## 第十二章：无障碍与引导

### 12.1 Onboarding 策略（零打断）

因为 App 是完全被动的，无需传统 Onboarding 引导页。

**策略：** toasts / 微交互动画，不弹窗、不浮层、不打断

| 时机 | 引导内容 | 形式 |

| 首次完成一次轨迹 | toast "1 条轨迹已保存，可在记录 Tab 中查看" | 2 秒后自动消失 |
| 首次打开记录 Tab（数据<3条） | 日历区域显示微弱提示"点击日期查看当日足迹" | 灰色小字，不抢眼 |

| 数据不足时的分析 Tab | 图表区灰色，显示等待文案 | 不展示空白图表 |

### 12.2 定位权限引导（静默优先）

**唯一的一次系统权限请求：**
- 首次打开 App → 浏览器/系统弹出定位权限对话框（原生，无 UI 可定制）
- 界面不需要额外文字提示（用户授权后直接进入地图界面）

**用户拒绝定位后：**
- 不重复弹权限窗
- 不弹引导提示
- 首页显示"位置未授权"
- 不记录轨迹
- 无任何 UI 变化（App 保持可用但无数据）

**用户手动开启定位后：**
- 系统返回通知 → 自动恢复记录
